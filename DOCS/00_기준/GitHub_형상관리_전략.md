# GitHub 형상관리 전략

## 원칙
- AI-PY 저장소는 GitHub Flow를 사용합니다.
- 기본 브랜치는 `main`입니다.
- 모든 작업은 `main`에서 분기한 작업 브랜치에서 수행합니다.
- 직접 `main`에 push 하지 않습니다.
- 모든 변경은 PR로만 `main`에 merge 합니다.
- 원칙적으로 `1기능 = 1브랜치 = 1PR`로 운영합니다.

## 커밋/PR 언어 규칙
- 커밋 메시지는 한글로 작성합니다.
- PR 제목/본문/리뷰 코멘트도 한글로 작성합니다.

## 커밋 전략
- AI 체인, API, 프롬프트, 문서 변경은 가능한 한 분리 커밋합니다.
- 하나의 커밋은 하나의 의도만 담습니다.
- `.env`, API key, 개인 토큰 파일은 절대 커밋하지 않습니다.

## 작업 단계 규칙
1. 명령 수신
2. 범위 잠금
3. 계획 수립
4. 구현
5. 테스트
6. Oracle 리뷰
7. 커밋/푸시
8. PR 생성
9. 리뷰 반영
10. merge 후 브랜치 정리
